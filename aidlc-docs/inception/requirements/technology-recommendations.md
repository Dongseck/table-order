# Technology Stack Recommendations

테이블오더 MVP 프로젝트에 대한 기술 스택 비교 및 추천입니다.

---

## Q2. 프론트엔드 프레임워크

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) React (Vite + TS) | 최대 생태계, 컴포넌트 라이브러리 풍부, TypeScript 지원 우수, 백엔드와 TS 통일 | 상태관리 라이브러리 별도 선택 필요 | ★★★★★ |
| B) Vue.js (Vite + TS) | 학습 곡선 낮음, 단일 파일 컴포넌트, 직관적 API | React 대비 생태계 작음, UI 라이브러리 선택지 적음 | ★★★★ |
| C) Next.js | SSR 기본 제공, 풀스택 가능 | 태블릿 SPA에 SSR 불필요, 설정 복잡 | ★★★ |
| D) Angular | 대규모 엔터프라이즈에 적합, 올인원 프레임워크 | 무거움, 학습 곡선 높음, MVP에 과도 | ★★ |

**추천: A) React (Vite + TypeScript)**
- 이유: Node.js 백엔드와 TypeScript 스택 통일, 터치 친화적 UI 컴포넌트 라이브러리 풍부(MUI, Ant Design 등), SSE 연동 라이브러리 다수 존재

## Q3. 데이터베이스

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) PostgreSQL | 트랜잭션 강력, JSON 지원, 관계형 데이터에 최적, 무료 | 약간의 학습 곡선 | ★★★★★ |
| B) MySQL | 가벼움, 널리 사용됨 | JSON/고급 기능 PostgreSQL 대비 부족 | ★★★★ |
| C) MongoDB | 유연한 스키마, 빠른 프로토타이핑 | 주문-메뉴-테이블 간 관계 표현 불편, 트랜잭션 제한적 | ★★★ |
| D) DynamoDB | 무한 확장성, AWS 네이티브 | 관계형 쿼리 어려움, 로컬 개발 불편, 비용 예측 어려움 | ★★ |

**추천: A) PostgreSQL**
- 이유: 매장-테이블-주문-메뉴 간 관계형 데이터 구조에 최적, 주문 생성 시 트랜잭션 보장, Node.js에서 Prisma ORM으로 쉽게 연동, Docker로 로컬 구동 간편

## Q4. 배포 환경

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) AWS (EC2, RDS 등) | 프로덕션 준비, 확장 가능 | 초기 설정 복잡, 비용 발생 | ★★★ |
| B) 로컬/온프레미스 | 비용 없음, 단순 | 외부 접근 어려움, 확장 불가 | ★★ |
| C) Docker + Compose | 로컬 개발 간편, 이식성 높음, 추후 클라우드 전환 용이 | 프로덕션 배포 시 추가 작업 필요 | ★★★★★ |
| D) Serverless | 비용 효율적, 자동 확장 | SSE 지원 어려움, 콜드스타트, 로컬 개발 복잡 | ★★ |

**추천: C) Docker + Docker Compose**
- 이유: MVP 단계에서 로컬 개발 환경을 빠르게 구축, PostgreSQL/Node.js/React를 컨테이너로 통합 관리, 추후 AWS/GCP 등 어디든 이전 가능, SSE 실시간 통신에 제약 없음

---

## Q5. 프론트엔드 프로젝트 구조

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) 단일 프로젝트 (라우팅 분리) | 개발 속도 빠름, 공통 코드 공유 용이, 단일 빌드 | 번들 크기 커질 수 있음, 코드 경계 불명확 | ★★★★★ |
| B) 별도 프로젝트 2개 | 완전 독립 배포, 명확한 분리 | 공통 코드 중복, 관리 포인트 2배 | ★★★ |
| C) 모노레포 | 공유 + 독립 배포 모두 가능 | 설정 복잡(Turborepo/Nx 필요), MVP에 과도 | ★★★ |

**추천: A) 단일 프로젝트에서 라우팅으로 분리**
- 이유: MVP에서 가장 빠른 개발 속도, TypeScript 타입/API 클라이언트/컴포넌트 공유, `/customer/*`와 `/admin/*` 라우트로 자연스러운 분리, 코드 스플리팅으로 번들 최적화 가능

## Q6. 메뉴 이미지 관리

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) 외부 이미지 URL 입력 | 가장 단순, 인프라 불필요 | 관리자가 URL 직접 준비해야 함 | ★★★★★ |
| B) 서버 파일 업로드 | 관리자 편의, 단일 서버 관리 | 디스크 용량 관리, 백업 복잡 | ★★★★ |
| C) S3 클라우드 스토리지 | 확장성 우수, CDN 연동 | AWS 설정 필요, MVP에 과도 | ★★★ |

**추천: A) 외부 이미지 URL 직접 입력**
- 이유: MVP에서 파일 업로드 인프라 구축은 과도, 요구사항에서도 이미지 리사이징/최적화를 제외했으므로 URL 입력이 적합, 추후 업로드 기능 추가 가능

---

## Q7. 예상 규모

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) 소규모 (단일 매장, 10개 이하) | 설계 단순 | 너무 작으면 실용성 부족 | ★★★ |
| B) 중규모 (단일 매장, 10~50개) | 현실적 규모, 적절한 복잡도 | - | ★★★★★ |
| C) 대규모 (다중 매장) | 확장성 고려 | MVP 범위 초과, 멀티테넌시 복잡 | ★★ |

**추천: B) 중규모 - 단일 매장, 테이블 10~50개**
- 이유: 실제 음식점 규모와 일치, 단일 매장 기준으로 설계하면 복잡도 관리 용이, 다중 매장은 추후 확장 가능

---

## Q8. 아키텍처 패턴

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) 모놀리식 | 가장 단순, 빠른 개발 | 코드가 커지면 유지보수 어려움 | ★★★★ |
| B) 모듈형 모놀리스 | 단일 배포 + 도메인 분리, 확장 준비 | 약간의 초기 설계 비용 | ★★★★★ |
| C) 마이크로서비스 | 독립 확장/배포 | MVP에 과도, 인프라 복잡, 개발 속도 저하 | ★★ |

**추천: B) 모듈형 모놀리스**
- 이유: 하나의 서버로 배포하되, 도메인별(auth, store, menu, order, table) 모듈로 분리하여 코드 정리. 마이크로서비스의 복잡성 없이 깨끗한 구조 확보, 추후 필요 시 모듈을 독립 서비스로 분리 가능

---

## Q9. 테스트 전략

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) 기본 (단위 테스트만) | 최소 비용 | 통합 이슈 놓칠 수 있음 | ★★★ |
| B) 표준 (단위 + API 통합) | 균형 잡힌 품질 보장, API 동작 검증 | 약간의 추가 시간 | ★★★★★ |
| C) 포괄적 (단위 + 통합 + E2E) | 최대 커버리지 | E2E 유지보수 비용 높음, MVP에 과도 | ★★★ |

**추천: B) 표준 - 단위 테스트 + API 통합 테스트**
- 이유: 핵심 비즈니스 로직(주문 금액 계산, 세션 관리 등)을 단위 테스트로, API 엔드포인트를 통합 테스트로 검증. MVP에 적절한 균형

---

## Q10. 보안 확장 규칙

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) Yes (전체 적용) | 프로덕션급 보안 보장 | 개발 속도 저하, MVP에 과도할 수 있음 | ★★★ |
| B) No (건너뛰기) | 빠른 개발 | 기본 보안은 요구사항에 이미 포함됨 | ★★★★★ |

**추천: B) No - 건너뛰기**
- 이유: 요구사항에 이미 기본 보안(bcrypt 해싱, JWT, 로그인 시도 제한)이 명시되어 있으므로 이를 구현하면 충분. 전체 보안 확장은 프로덕션 배포 전 별도 보안 리뷰 단계에서 적용 권장

---

## Q11. Property-Based Testing

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|--------|
| A) Yes (전체 적용) | 비즈니스 로직 버그 사전 발견 | 학습 곡선, 테스트 작성 시간 증가 | ★★ |
| B) Partial (순수 함수만) | 핵심 계산 로직만 검증 | 제한적 효과 | ★★★ |
| C) No (건너뛰기) | 빠른 개발, CRUD 위주 프로젝트에 적합 | - | ★★★★★ |

**추천: C) No - 건너뛰기**
- 이유: 테이블오더는 주로 CRUD 작업(메뉴 조회, 주문 생성, 상태 변경)으로 구성. 복잡한 알고리즘이나 데이터 변환이 거의 없어 PBT의 이점이 제한적. 표준 단위/통합 테스트로 충분
